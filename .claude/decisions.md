# Decision Log

프로젝트의 주요 기술 결정을 기록한다.
새 세션에서 Claude가 이전 결정을 인지하고 일관된 방향을 유지하도록 세션마다 자동 로드됨.

## 기록 시점

아래 상황이 발생하면 **반드시** 이 파일에 기록한다:
- 계획서에 없던 기능 추가 또는 기존 기능 삭제
- 기능 연기(defer)도 삭제와 동일하게 기록 ("나중에 하자" = 스코프 변경)
- 외부 서비스·API 확정 또는 변경 (결제, 인증, 메일 등)
- 기술 스택 변경 (라이브러리, DB, 프레임워크 교체)
- 설계 패턴 변경 (라우팅 구조, 상태 관리 방식, 데이터 흐름 등)
- 계획서 대비 우선순위 변경 (P0↔P1 교체, 스코프 축소 등)

## 놓치기 쉬운 기록 상황

위 목록에 해당하지만 **다른 작업으로 포장**되어 놓치기 쉬운 경우:
- **성능 수정으로 위장된 설계 변경**: "느려서 서버 컴포넌트로 바꿔" → CSR→SSR 전환은 데이터 페칭 전략 변경이다
- **디버깅 중 라이브러리 교체**: 에러 해결 목적이라도 package.json의 의존성이 바뀌면 기록
- **한 곳 수정이지만 전체 패턴에 영향**: 한 페이지의 변경이 프로젝트 전체 패턴의 선례가 되는 경우

## 자체 점검 규칙

- **package.json 변경 시**: dependencies가 추가·제거·교체되면 기록 필요 여부를 자체 점검
- **세션 종료 전**: "이번 세션에서 계획서 대비 달라진 점이 있는가?" 자체 점검 후 미기록 결정이 있으면 소급 기록

## 기록 포맷

```
### YYYY-MM-DD: 한 줄 요약
- **결정**: 무엇을 어떻게 하기로 했는가
- **이유**: 왜 이 선택을 했는가 (대안 대비 근거)
- **영향**: 이 결정이 다른 부분에 미치는 영향
- **관련 파일**: 변경된 주요 파일 경로
```
